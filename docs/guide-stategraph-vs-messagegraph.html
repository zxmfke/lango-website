StateGraph vs MessageGraph

LangGraphGo 提供了两种主要的图构建方式：StateGraph 和 MessageGraph（或 MessagesStateGraph）。选择哪一种取决于你的应用需求和状态管理的复杂性。

1. StateGraph

背景与功能

StateGraph 是最通用的图类型。它允许你完全自定义状态的 Schema（结构）。你可以定义任意的 Go 结构体或 Map 作为状态，并为每个字段指定特定的 Reducer（合并函数）。

适用场景

  • 需要管理复杂的结构化数据（如订单信息、用户信息、多步骤任务进度）。
  • 状态不仅仅是消息列表，还包含其他元数据。
  • 需要对状态更新进行细粒度控制。

代码示例

// 定义自定义状态结构
type AgentState struct {
    Input    string
    Messages []llms.MessageContent
    Steps    []string
}

// 创建 StateGraph
g := graph.NewStateGraph(graph.StateSchema{
    "messages": graph.AppendReducer,
    "steps":    graph.AppendReducer,
})

2. MessageGraph

背景与功能

MessageGraph（以及其增强版 MessagesStateGraph）是 StateGraph 的一种特化形式，专门为聊天应用设计。它的状态 Schema 被预定义为一个消息列表（[]llms.MessageContent）。

适用场景

  • 构建纯聊天机器人或对话代理。
  • 状态只需要包含对话历史。
  • 希望快速上手，无需定义复杂的 Schema。

代码示例

// 创建 MessageGraph
// 状态默认为 []llms.MessageContent
g := graph.NewMessageGraph()

// 节点直接返回消息，自动追加到历史中
g.AddNode("chatbot", func(ctx context.Context, state interface{}) (interface{}, error) {
    return []llms.MessageContent{...}, nil
})

3. 对比总结

特性 StateGraph MessageGraph
状态灵活性 高（完全自定义） 低（仅限消息列表）
上手难度 中 低
典型用例 复杂 Agent、业务流程 聊天机器人
